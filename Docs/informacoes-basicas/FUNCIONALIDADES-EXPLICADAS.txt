PIROFAFE — Funcionalidades explicadas (linguagem simples)
==========================================================

Este ficheiro explica, de forma simples, as principais “funcionalidades” e conceitos do projeto.
Não precisas de saber programar para entender o essencial.


1) Tem backend + frontend
------------------------
O que é:
- Pensa na aplicação como um restaurante:
  - O FRONTEND é o “menu e o empregado” (o que tu vês no browser e onde clicas).
  - O BACKEND é a “cozinha” (onde as regras e os dados vivem).

Como funciona aqui:
- FRONTEND (pasta "apps/web/"):
  - É feito em Next.js (um tipo de app web moderna).
  - Mostra páginas (clientes, encomendas, paióis, etc.) e faz pedidos ao backend.
- BACKEND (na raiz do repositório):
  - É uma API em ASP.NET Core (um servidor).
  - Guarda/consulta dados numa base de dados e aplica regras (ex.: validar permissões, preparar encomendas).


2) Tem autenticação por tokens (JWT) com refresh token
------------------------------------------------------
O que é:
- Autenticação é “provar quem tu és” (como fazer login).
- Em vez de usar “sessão” normal (cookie apenas), o projeto usa TOKENS:
  - Access token (JWT): um “bilhete” que prova que estás autenticado durante algum tempo.
  - Refresh token: um “bilhete de renovação” para pedir um novo access token quando o antigo expira.

Como funciona aqui (resumo do fluxo):
- Fazes login com email + password.
- O backend devolve:
  - token (JWT) + refreshToken + data de expiração.
- O frontend guarda esses tokens (para a sessão atual) e envia o JWT em cada pedido protegido:
  - Header: Authorization: Bearer <token>
- Quando o JWT está a acabar, o frontend pede ao backend para renovar:
  - /api/auth/refresh (usa o refreshToken)
- “Rotação” do refresh token:
  - Quando renova, o backend pode dar um refresh token novo e invalidar o anterior.
  - Isto melhora a segurança (se alguém roubar um refresh antigo, pode já não servir).
- Logout:
  - O frontend pede /api/auth/logout para o backend marcar o refresh token como revogado (inutilizado).


3) Tem autorização por roles e políticas (permissões)
-----------------------------------------------------
O que é:
- Autorização é “o que tens permissão para fazer”.
- Role (papel) é o teu “cargo” no sistema. Exemplos canónicos:
  - **Admin**, **Gestor** (antigo nome “Técnico”), **Comercial**, **Armazém**
- Política é uma regra que diz:
  - “Para entrar nesta área / fazer esta ação, precisas de X roles”.

Como funciona aqui:
- Mesmo que estejas autenticado (com token), alguns endpoints só funcionam se tiveres role/permissão certa.
- O endpoint /api/auth/me devolve também as tuas “permissions” (permissões em texto),
  para o frontend conseguir mostrar/esconder botões e menus de forma consistente com o backend.

Exemplo simples:
- Um utilizador pode “ver stock” mas não “gerir stock”.
- Um Admin pode ver e gerir tudo; outros veem apenas o que lhes é permitido.


4) Tem Swagger (só em Development)
----------------------------------
O que é:
- Swagger é uma página “de teste” da API:
  - Mostra a lista de endpoints.
  - Permite fazer chamadas e ver respostas sem precisar de criar um programa à parte.

Como funciona aqui:
- Em desenvolvimento, dá para abrir /swagger e testar login, clientes, encomendas, etc.
- Em produção, fica desligado para reduzir riscos (não expor informação técnica publicamente).


5) Tem testes (backend e frontend)
----------------------------------
O que é:
- Testes são “verificações automáticas” para garantir que algo continua a funcionar
  mesmo depois de alterações no código.

Como funciona aqui:
- Backend:
  - Projeto "Finalproj.Tests" com xUnit.
  - Testa regras importantes do domínio (ex.: preparação de encomendas e cálculo de stock disponível).
- Frontend:
  - Vitest + Testing Library:
    - Testes automáticos para componentes/lógica do frontend.
  - Playwright E2E:
    - Testes que simulam um utilizador a navegar e clicar (fim-a-fim).


6) Tem CI do frontend (automação ao subir código)
-------------------------------------------------
O que é:
- CI (Integração Contínua) é um “robô” que corre automaticamente verificações quando alguém faz push/PR.

Como funciona aqui (resumo):
- Sempre que há mudanças no frontend (apps/web/**), o CI corre:
  - typecheck (verificar tipos do TypeScript)
  - lint (regras de qualidade)
  - testes (unit/component)
  - build (garantir que compila)
  - testes E2E (Playwright)

Para que serve:
- Evita que uma alteração “parta” o frontend sem ninguém reparar.


7) Tem módulos de negócio principais (API)
------------------------------------------
O que é:
- A API está organizada por “áreas” (módulos), cada uma com endpoints.

Como funciona aqui (o que cada área faz):

7.1) Paióis / Armazém
- Serve para gerir armazéns (paióis), limites e ocupação.
- Guarda movimentos de stock:
  - Entradas (o que entrou)
  - Saídas (o que saiu)
- Pode ter documentos anexos (ex.: licenças).
- O acesso pode depender do papel do utilizador (quem pode ver/gerir que paiol).

7.2) Produtos
- Catálogo de produtos pirotécnicos.
- Guarda informação técnica (ex.: NEM por unidade, compatibilidades, etc.).

7.3) Encomendas
- Pedidos de clientes.
- Têm um ciclo de vida:
  - Pendente → Aceite → Em preparação → Concluída
  - (ou Rejeitada)
- “Reservas”:
  - Enquanto a encomenda está em certos estados, o stock fica “reservado” para essa encomenda.
- Preparação FIFO:
  - Ao preparar, o sistema tenta retirar stock por ordem de entrada (primeiro a entrar, primeiro a sair).

7.4) Serviços no terreno
- Representa um serviço/evento no terreno ligado a uma encomenda concluída.
- Inclui:
  - Equipa (quem vai)
  - Licenças (documentos)
  - Distâncias de segurança (registos)
  - Documentos extra

7.5) Clientes
- CRUD de clientes (criar, ver, editar, apagar).
- Pode incluir documentos e histórico de encomendas.

7.6) Funcionários
- CRUD de funcionários.
- Pode associar um funcionário a uma conta de utilizador (Identity).
- Documentos do funcionário (ex.: identificações/licenças).

7.7) Admin
- Gestão de utilizadores e roles.
- Operações administrativas (ex.: algumas ações de manutenção/backup).

7.8) Home / Perfil / Preferências
- Página inicial/estatísticas.
- Perfil do utilizador (nome, dados).
- Preferências (ex.: opções de UI/tema, dependendo do que estiver implementado).
- Alterar password.


8) Tem validação de dados (FluentValidation)
--------------------------------------------
O que é:
- Antes de gravar dados, o sistema verifica se estão corretos:
  - Campos obrigatórios, formatos, limites, regras específicas.

Como funciona aqui:
- No backend, há validadores para vários formulários/DTOs.
- Se algo estiver errado, a API responde com erro 400 e mensagens em português.


9) Tem gestão documental (upload/download)
------------------------------------------
O que é:
- Permite anexar ficheiros a entidades (paióis, serviços, clientes, funcionários).

Como funciona aqui:
- Upload:
  - O frontend envia ficheiro (multipart/form-data).
  - O backend valida (extensões/tamanho) e guarda em disco (wwwroot).
- Download/visualização:
  - O backend devolve o ficheiro quando pedes um endpoint de documento.

Nota importante (segurança):
- Normalmente, além de existir o ficheiro, o sistema também tem de confirmar se tens permissão
  para aceder ao documento (ex.: não dar documentos de outro paiol/serviço a quem não tem acesso).


10) Tem observabilidade básica (X-Correlation-Id)
-------------------------------------------------
O que é:
- Quando algo corre mal, é útil ter um “número de referência” para encontrar o erro nos logs.
- Correlation Id é esse identificador.

Como funciona aqui:
- Cada resposta inclui um header "X-Correlation-Id".
- Em caso de erro (ex.: 500), o backend pode incluir esse id no JSON.
- Assim, alguém que administra o servidor consegue procurar esse id nos logs e perceber o que aconteceu.


11) Tem backups automáticos da base de dados
--------------------------------------------
O que é:
- Um backup é uma “cópia de segurança” da base de dados.

Como funciona aqui:
- Existe um serviço em background que faz backup do SQL Server todos os dias a uma hora definida.
- Guarda ficheiros .bak no projeto/servidor (consoante configuração).
- Pode existir também uma forma manual (via admin) para forçar um backup quando necessário.

Para que serve:
- Recuperar dados em caso de falha, erro humano, corrupção ou necessidade de auditoria.


12) Tem CORS configurável (para o frontend comunicar com a API)
--------------------------------------------------------------
O que é:
- O browser tem regras de segurança que bloqueiam chamadas entre domínios/portas diferentes.
- CORS é a configuração que diz:
  - “Este site (frontend) pode chamar esta API (backend)”.

Como funciona aqui:
- Em desenvolvimento:
  - Aceita origens locais típicas (ex.: http://localhost:3000).
- Em produção:
  - Deve ser configurado com a(s) origem(ns) real(is) da app (ex.: https://app.seudominio.pt).


13) Tem “primeiro utilizador” (bootstrap do sistema)
----------------------------------------------------
O que é:
- Em sistemas novos, ainda não existe nenhum utilizador Admin para configurar tudo.
- “Primeiro utilizador” é um fluxo especial para iniciar o sistema.

Como funciona aqui:
- Existe um endpoint público para verificar se já há utilizadores.
- Se não houver:
  - Permite registar o primeiro utilizador.
  - Esse utilizador recebe a role Admin.
- Depois de existir pelo menos um utilizador:
  - Esse registo especial deixa de estar disponível (para evitar abusos).
